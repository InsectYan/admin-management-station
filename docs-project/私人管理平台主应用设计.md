## 私人管理平台基础框架设计报告

### 1. 报告概述

本报告为私人管理平台主应用的基础框架搭建提供完整的技术设计方案和开发指导。平台采用 React Web SPA 作为前端，Egg.js 作为 BFF 后端，Qiankun作为微前端容器，PostgreSQL作为数据库。报告分为两大部分：**设计方案**和**开发设计方案**，前者提供技术选型、架构设计、交互流程等宏观视角，后者提供具体开发步骤、代码示例和配置细节，确保开发团队能够根据报告直接实施代码开发。

报告详细涵盖了前端UI渲染、后端整体设计、后端细节设计、数据库设计、微前端架构设计等关键内容，特别关注了菜单与子应用的微前端整合、菜单数据的独立存储与管理、子应用的动态加载与卸载机制等核心需求。

### 2. 技术栈选型与优势分析

#### 2.1 前端技术栈：React Web SPA

**React**：作为前端 SPA 框架，组件化、声明式 UI，通过 Vite 构建，由 Nginx 容器托管静态资源。

**Qiankun**：微前端框架，支持任意技术栈子应用接入，提供样式隔离、JS 沙箱、资源预加载。

**Docker + Nginx**：前端构建产物打包为镜像，统一 HTTP 入口；子应用独立容器部署，Qiankun 通过 HTTP entry 加载。

**技术优势组合**：React SPA 与 Egg.js BFF 分离部署，Docker 保证环境一致；React 与 Qiankun 结合，菜单与子应用解耦，便于独立迭代。

#### 2.2 后端技术栈：Node.js + Egg.js

**Egg.js**：作为企业级Node.js框架，提供进程管理、插件系统、配置分环境管理等企业级能力，支持高并发和复杂业务场景。

**优势**：Egg.js内置Master-Worker多进程模型，天然支持多核CPU利用率最大化；提供完整的插件生态，如`egg-jwt`（认证）、`egg-orm`（数据库）等；通过 HTTP RESTful API 与前端通信，适合容器化部署。

#### 2.3 数据库：PostgreSQL

**优势**：支持复杂查询和事务，适合结构化数据存储；提供JSONB类型，便于存储菜单配置等半结构化数据；与TypeORM ORM工具兼容性良好，简化了数据库操作。

#### 2.4 微前端架构：Qiankun

**优势**：基于single-spa二次封装，提供开箱即用的微前端能力；支持技术栈无关，允许React、Vue等不同技术栈的子应用共存；提供样式隔离和JS沙箱机制，确保子应用间互不影响；支持动态加载和预加载，提升用户体验。

### 3. 设计方案

#### 3.1 整体架构设计

**架构分层**：

```
+---------------------------+
|         浏览器            |  用户通过 Web 访问平台
+---------------------------+
|    Nginx（主应用静态）     |  托管 React 基座构建产物
+---------------------------+
|          Qiankun           |  微前端容器，子应用加载/卸载/通信
+---------------------------+
|        主应用 React        |  菜单渲染、路由管理、子应用容器
+---------------------------+
|    子应用 SPA（独立容器）   |  每个一级菜单对应一个子应用
+---------------------------+
|   Egg.js BFF（API 容器）   |  菜单 API、鉴权、权限管理
+---------------------------+
|  PostgreSQL / Redis 容器   |  持久化与缓存
+---------------------------+
```

**Docker 部署优势**：前后端、数据库、缓存均以容器运行。Nginx 托管主应用与子应用静态资源并反向代理 API；详见 [部署与Docker方案.md](./部署与Docker方案.md)。

#### 3.2 微前端架构设计

**Qiankun与React集成**：主应用使用 React Web SPA 作为基座，通过 Qiankun 实现子应用动态加载与管理。

**子应用加载模式**：采用"路由匹配"模式，每个子应用对应一个一级菜单，通过路由前缀匹配加载。

**子应用容器设计**：在右侧内容区域设置一个统一的子应用容器，所有子应用都挂载到该容器中，通过路由变化动态切换加载的子应用。

**子应用通信机制**：采用props传递和全局事件总线两种通信方式，props用于主应用向子应用传递必要参数，全局事件总线用于子应用间或子应用与主应用间的事件通信。

#### 3.3 菜单系统设计

**菜单数据结构**：菜单采用树形结构存储，每个菜单项包含以下信息：

- id: 唯一标识符
- name: 菜单名称
- parent_id: 父菜单ID（根菜单为NULL）
- route_prefix: 路由前缀（如'media1'对应'/media/media1'）
- microapp_name: 子应用名称（需与Qiankun注册名称一致）
- status: 状态（启用/禁用）
- order: 排序
- icon: 图标路径

**菜单与子应用的映射关系**：一级菜单项直接映射为子应用，每个一级菜单对应一个子应用。菜单数据与子应用代码完全解耦，子应用的加载、卸载仅通过菜单数据的更新实现。

**菜单加载流程**：

1. 主应用启动时，通过 HTTP API 获取菜单配置数据
2. 根据菜单数据动态渲染左侧菜单栏
3. 将一级菜单项注册为Qiankun子应用
4. 路由变化时，根据当前路由匹配对应的子应用并加载

**菜单与子应用的通信**：菜单项点击事件触发路由变化，Qiankun根据路由前缀匹配子应用并加载。子应用可通过props获取菜单项的元数据。

#### 3.4 数据库设计

**菜单表设计**：

```sql
CREATE TABLE menu_items (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  parent_id INTEGER REFERENCES menu_items(id),
  route_prefix VARCHAR(50) NOT NULL,
  microapp_name VARCHAR(50) NOT NULL,
  status VARCHAR(20) DEFAULT 'enabled',
  order INTEGER DEFAULT 0,
  icon VARCHAR(200),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**索引设计**：

```sql
CREATE INDEX idx_parent_id ON menu_items(parent_id);
CREATE INDEX idx_route_prefix ON menu_items(route_prefix);
CREATE INDEX idx_status ON menu_items(status);
```

**数据示例**：

```sql

VALUES
  ('媒体管理', NULL, 'media', 'media-app', 'enabled', 1, 'icon-media'),
  ('视频', 1, 'media/video', 'video-app', 'enabled', 1, 'icon-video'),
  ('音频', 1, 'media-audio', 'audio-app', 'enabled', 2, 'icon-audio'),
```

**菜单表结构说明**：菜单表采用树形结构设计，通过parent_id字段实现层级关系。一级菜单项的parent_id为NULL，代表根节点。每个一级菜单项对应一个子应用，通过microapp_name字段与Qiankun注册的子应用名称保持一致，实现菜单与子应用的动态绑定。

#### 3.5 后端服务设计

**API接口设计**：

- GET /api/menus：获取所有菜单项
- GET /api/menus/root：获取一级菜单项
- POST /api/menus：添加新菜单项
- PUT /api/menus/:id：更新菜单项
- DELETE /api/menus/:id：删除菜单项

**权限管理**：菜单管理API需要严格的权限控制，只有管理员角色才能调用添加、更新、删除接口。通过Egg.js的中间件实现JWT验证和RBAC权限模型。

**菜单缓存机制**：为提高性能，菜单数据在后端采用Redis缓存，前端采用状态管理工具（如Redux）缓存，避免频繁访问数据库。

**菜单变更监听**：主应用可监听菜单数据变更事件，当菜单数据发生变化时，自动重新加载子应用配置。

#### 3.6 子应用管理设计

**子应用注册**：主应用启动时，从菜单API获取一级菜单数据，动态注册子应用到Qiankun框架中。

**子应用卸载**：管理员通过菜单管理界面禁用某个一级菜单时，主应用调用Qiankun的unregisterMicroApp方法卸载对应子应用。

**子应用热更新**：支持子应用的独立更新，无需重启主应用。主应用可通过菜单API获取子应用的最新版本信息，实现无缝更新。

**子应用状态管理**：主应用维护子应用的加载状态，包括已加载、未加载、加载中等，确保用户体验流畅。

#### 3.7 安全与权限设计

**菜单数据安全**：菜单数据仅由主应用管理，子应用无直接访问权限，通过API接口获取菜单配置。

**权限控制**：菜单API接口采用RBAC权限模型，通过JWT令牌验证用户权限，确保只有管理员才能修改菜单结构。

**子应用沙箱隔离**：Qiankun框架为每个子应用提供JS沙箱隔离，防止子应用间的全局变量污染和样式冲突。

**资源加载安全**：子应用资源通过HTTP协议加载，子应用与静态资源均通过 HTTP/HTTPS 加载。

#### 3.8 路由设计

**主应用路由**：使用React Router作为主应用的路由框架，定义以下主要路由：

- /:appKey/*：匹配子应用路由，其中appKey为一级菜单的route_prefix
- /:path(*)：兜底路由，处理未匹配的路径

**子应用路由**：每个子应用内部使用自己的路由框架（如React Router或Vue Router），但必须设置正确的basename，确保路由与主应用路由隔离。

**路由与菜单联动**：主应用路由与菜单数据联动，当用户点击菜单项时，主应用更新URL路径，触发Qiankun加载对应子应用。

#### 3.9 状态管理设计

**菜单状态管理**：使用Redux或Context API管理菜单状态，包括菜单数据、当前选中菜单项、菜单加载状态等。

**子应用状态管理**：每个子应用独立管理自身状态，主应用不直接干预子应用内部状态。子应用间需要共享的状态通过Qiankun的全局状态机制管理。

**全局状态共享**：主应用维护全局状态（如用户登录信息、主题设置等），子应用可通过Qiankun的props或全局状态API访问这些信息。

**菜单模型开发**：

```javascript
// apps/main-backend/model/Menu.js
const { Model } = require(' egg-mysql');

class Menu extends Model {
  static table = 'menu_items';

  // 静态方法获取菜单树
  static async getTree() {
    const all = await this.query('SELECT * FROM menu_items ORDER BY parent_id, order');
    const tree = [];
    const map = new Map();

    all.forEach(item => {
      map.set(item.id, item);
      if (item.parent_id) {
        const parent = map.get(item.parent_id);
        if (parent) {
          if (!parent.children) {
            parent.children = [];
          }
          parent.children.push(item);
        }
      } else {
        tree.push(item);
      }
    });

    return tree;
  }
}

module.exports = Menu;
```

**菜单管理界面开发**：提供菜单管理界面，供管理员添加、更新、删除菜单项。

```javascript
// frontend/src/components/MenuManager.jsx
import React, { useEffect, useState } from 'react';
import { nanoid } from 'nanoid';

function MenuManager() {
  const [ menus, setMen us] = useState([]);
  const [ loading, setLoading ] = useState(true);

  // 获取菜单数据
  useEffect(() => {
    const fetch menus = async () => {
      try {
        const response = await fetch('http://localhost:7001/api/menus');
        const data = await response.json();
        setMen us(data);
        setLoading(false);
      } catch (err) {
        console.error('获取菜单失败:', err);
        setLoading(false);
      }
    };

    fetch menus();
  }, []);

  // 添加菜单项
  const handleAddMenu = async (menuData) => {
    try {
      const response = await fetch('http://localhost:7001/api/menus', {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'Authorization': 'Bearer ' + JWT令牌
        },
        body: JSON.stringify(menuData)
      });

      if (response.ok) {
        const newMenu = await response.json();
        setMen us([...men us, newMenu]);
      }
    } catch (err) {
      console.error('添加菜单失败:', err);
    }
  };

  // 更新菜单项
  const handleUpdateMenu = async (menuId, menuData) => {
    try {
      const response = await fetch(`http://localhost:7001/api/menus/${menuId}`, {
        method: 'PUT',
        headers: {
          'Content-Type': 'application/json',
          'Authorization': 'Bearer ' + JWT令牌
        },
        body: JSON.stringify(menuData)
      });

      if (response.ok) {
        const updatedMenu = await response.json();
        const newMen us = men us.map((menu) =>
          menu.id === menuId ? updatedMenu : menu
        );
        setMen us(newMen us);
      }
    } catch (err) {
      console.error('更新菜单失败:', err);
    }
  };

  // 删除菜单项
  const handleDeleteMenu = async (menuId) => {
    try {
      const response = await fetch(`http://localhost:7001/api/menus/${menuId}`, {
        method: 'DELETE',
        headers: {
          'Authorization': 'Bearer ' + JWT令牌
        }
      });

      if (response.ok) {
        const newMen us = men us.filter((menu) => menu.id !== menuId);
        setMen us(newMen us);
      }
    } catch (err) {
      console.error('删除菜单失败:', err);
    }
  };

  return (
    <div className="menu-manager">
      {loading ? (
        <div className="loading">加载中...</div>
      ) : (
        <div className="menu-tree">
          {renderMenuTree(menus)}
        </div>
      )}
    </div>
  );
}

// 递归渲染菜单树
function renderMenuTree(menus, level = 0) {
  return menus.map((menu) => (
    <div key={nanoid()} className="menu-node" style={{ indentation: level * 20 + 'px' }}>
      <div className="name">
        {menu.name}
      </div>
      <div className="actions">
        <button onClick={() => handleUpdateMenu(menu.id, menu)}>更新</button>
        <button onClick={() => handleDeleteMenu(menu.id)}>删除</button>
      </div>
      {menu.children && renderMenuTree(menu.children, level + 1)}
    </div>
  ));
}
```

#### 4.6 AI Agent集成设计

**AI Agent架构**：采用模块化设计，将不同AI能力封装为独立的Agent，通过统一的接口与平台交互。

**Agent通信机制**：使用WebSocket或HTTP长连接与Agent服务通信，确保低延迟和高可靠性。

**Agent集成方式**：可选择以下任意一种方式集成AI Agent：

1. **作为子应用集成**：将AI Agent功能作为一级菜单项，通过子应用方式加载
2. **作为主应用功能集成**：在主应用中直接集成AI Agent功能，与菜单系统深度集成
3. **作为独立服务集成**：将AI Agent作为独立服务部署，通过API与平台交互

**推荐集成方案**：基于平台的微前端架构和AI Agent的灵活性需求，推荐采用**方式1和方式3结合**的方案：

- 基础AI能力（如搜索、智能推荐）作为子应用集成，提供菜单入口
- 复杂AI任务（如图像处理、自然语言处理）作为独立服务，通过API调用

#### 5.4 HTTP 资源加载

**实现原理**：主应用与子应用构建产物均由 Nginx 容器通过 HTTP 提供，Qiankun entry 指向 `/subapps/{name}/`。

**关键配置**：

```javascript
// apps/main-backend/config/config.default.js
module.exports = {
  // 其他配置...

  // 静态资源配置
  static: {
    dir: path.join(__dirname, '..', 'frontend', 'dist'),
    maxAge: 31536000, // 一年
    enable: true
  }
};
```

**优势**：容器化部署下路径统一，子应用可独立扩缩容。

#### 5.5 子应用独立更新与热替换

**实现原理**：子应用可独立更新，无需重启主应用。通过菜单API获取最新子应用配置，动态更新Qiankun子应用配置。

**关键代码**：

```javascript
// 子应用热更新
export async function updateSubApp(name, entry) {
  try {
    // 更新子应用配置
    await updateMicroApp(name, {
      entry: entry
    });

    // 重新加载子应用
    await reloadMicroApp(name);
  } catch (err) {
    console.error('更新子应用失败:', err);
    throw err;
  }
}
```

**优势**：支持子应用的独立更新和热替换，提高了系统的可维护性和用户体验。

### 8. 总结与建议

**系统优势**：
- **容器化部署**：Docker Compose 一键启动全栈，浏览器访问无需安装客户端
- **微前端架构**：Qiankun 支持子应用独立开发、构建为镜像、独立扩缩容
- **菜单与子应用解耦**：菜单数据驱动子应用注册，降低维护成本
- **安全可靠**：JWT + RBAC；Qiankun 沙箱隔离子应用
- **高性能**：PostgreSQL + Redis 缓存；Egg.js 多进程模型

**开发建议**：
- **遵循约定优于配置**：充分利用 Egg.js 与 React 社区约定
- **优先使用 HTTP API**：前后端仅通过 RESTful API 通信
- **启用严格样式隔离**：Qiankun `strictStyleIsolation`
- **合理使用缓存**：浏览器 localStorage + Redis 缓存菜单
- **Docker 化子应用**：每个子应用独立 Dockerfile，见 [部署与Docker方案.md](./部署与Docker方案.md)
- **定期更新依赖**：React、Qiankun、Egg.js 等核心依赖

**后续开发方向**：

- **完善权限管理**：扩展RBAC模型，实现更细粒度的权限控制
- **增加多语言支持**：为系统添加多语言支持，适应国际化需求
- **完善 CI/CD**：镜像构建、迁移与滚动发布流水线
